# ID Generation Strategy Guide

## Overview
The database uses a **hybrid ID management approach**:
- **User-visible IDs** (NV, KH, MUO, THU, BC): Generated by Backend as formatted strings
- **System IDs** (IDVatPham, IDDonMuaHang, etc.): Auto-increment INT in database
- **Membership IDs** (MaThe): Manual INT entry by user/backend

---

## 1. Employee ID (NV***)
**Table:** `NhanVien`  
**Column:** `IDNhanVien` (VARCHAR(20) PRIMARY KEY)  
**Format:** `NV` + 3-digit number (NV001, NV002, NV003...)

### Backend Logic:
```python
def generate_next_id(prefix='NV', table='NhanVien', column='IDNhanVien'):
    # Get all existing IDs starting with prefix
    SELECT IDNhanVien FROM NhanVien WHERE IDNhanVien LIKE 'NV%'
    # Extract max number and increment
    # Return formatted ID: NV + (max_num + 1).zfill(3)
```

### API Endpoint:
- **POST** `/api/employees`
  - **Body:** `{ "CMND_CCCD": "...", "Ho": "...", "Ten": "...", ... }`
  - **Response:** `{ "IDNhanVien": "NV001", ... }`
  - Backend automatically generates `IDNhanVien`

## 7. Membership Card Number (INT)
**Table:** `TheThanhVien`  
**Column:** `MaThe` (INT)  
**Format:** Simple integer (1, 2, 3...)

### Backend Logic:
```python
def get_next_membership_number():
    SELECT MAX(CAST(MaThe AS UNSIGNED)) FROM TheThanhVien
    return max_num + 1
```

### API Endpoint:
- **GET** `/api/membership/next-number`
  - **Response:** `{ "nextNumber": 4 }`
  - User/Admin then manually enters this number when creating membership card

## Performance Benefits

### 1. Flexible Primary Keys
- VARCHAR(20) for NV, KH, MUO, THU allows Backend to enforce custom rules
- Year-based IDs possible: NV24-001 (for 2024)
- Department-based IDs possible: NV-SALES-001

### 2. Optimized Searching
- Indexes on `IDNhanVien`, `IDKhachHang` columns
- Backend can efficiently use `SELECT MAX()` queries to find next numbers
- Fast lookups for user-visible IDs

### 3. Compatibility
- INT `AUTO_INCREMENT` for internal system operations
- VARCHAR for user-facing IDs for business logic flexibility

---

## Migration Notes

When the system scales:
1. **High Volume Employees:** Consider time-based IDs (NV24-001, NV25-001)
2. **User-Facing Reports:** INT IDs are hidden; only formatted strings displayed
3. **Integration:** External systems can call `/api/employees/<id>` with formatted ID

---
